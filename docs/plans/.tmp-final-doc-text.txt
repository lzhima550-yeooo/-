四季夏木
Summer Wood
校园植保健康智能社区
四季夏木（Summer Wood） 是一套面向校园园艺与家庭种植场景的植物健康智能助手。项目并非单一的“拍照识别工具”或“聊天页面”，而是围绕真实问题处理路径构建的完整应用：用户从上传图片开始，系统完成病虫害或生理异常识别、风险等级计算、治理建议生成、图鉴知识检索、人格化讲解、社区协作与后续复查追踪，最终形成一条可重复使用、可持续沉淀的知识闭环。
研发模式上，项目采用 Vibe Coding。 前后端页面、接口、数据模型与测试脚本由 Codex 驱动快速生成和迭代；设计层则由 Stitch 原型语言、Design Tokens 与 UI/UX 方法论加以约束；工程层以任务对象、统一 BFF、队列、RLS 权限、日志与部署链路支撑，确保“快”不等于“乱”，生成能力最终能收束为可交付的系统。

一、项目总述与定位
一句话定义
四季夏木不是一个“识别一下就结束”的工具，也不是一个只有对话框的 AI 页面，而是一套围绕校园真实植物问题构建的植物健康闭环系统：它能够识别、解释、给出行动、提供证据、生成角色化讲解，并把用户的求助与经验沉淀为可复用的知识。
项目的核心场景来自校园园艺、劳动教育、自然教育、社团实践与家庭种植的真实需求。在这些场景中，植物出现病斑、虫咬、黄化、卷叶、霉层或异常生长时，用户最缺的不是“一个名字”，而是“下一步应该怎么做”。传统识别工具往往停留在标签返回，无法把结果转化为可执行操作；传统知识页面内容繁杂、入口深、阅读负担大；普通问答社区又存在响应慢、内容质量参差、难以沉淀的问题。
四季夏木正是围绕这一断点建立产品逻辑。系统将“识别—解释—行动—追踪—共建”设计为连续链条，让用户从上传图片开始，就能够沿着系统给出的路径完成诊断理解、图鉴查证、治理执行、灵化学习与社区协作，而不是在多个工具之间来回切换。
与早期概念版本不同，当前完成态已经不再依赖 Mock 结果或演示链路。项目后端已经接入真实图像识别 API、真实对话 API 与完整生图服务；图鉴端接入全文检索、来源索引、治理模板与相关推荐；灵化端接入夏木提示词编排器、流式对话、种类驱动角色包与长期记忆；系统层面完成了任务队列、风险等级、行动卡片、消息通知、统一状态视图与埋点分析。因此，这份介绍书描述的不是“未来想做什么”，而是“当前完成版如何运行、如何组织、为什么成立”。
从答辩角度看，四季夏木的价值在于它同时具备产品完整度、工程可信度与教育表达力。它既能展示前端页面与交互设计，也能展示真实 AI 接入、数据治理、任务调度、生成式工作流与长期记忆机制；既能解决现实问题，也能通过人格化表达降低硬知识门槛，使系统更适合校园环境中的青少年用户。
（一）目标用户与使用场景
校园绿化管理与劳动教育场景：用于观察校园植物健康状态、辅助社团成员处理常见病虫害。
自然教育与科普实践场景：用更有参与感的方式把病虫害知识、生态关系、防治思路转化为可理解内容。
家庭园艺场景：帮助非专业用户完成从问题发现、症状理解到处理执行的全过程。
（二）系统完成态的核心判断
判断一个 AI 植物助手是否真正“完成”，不在于它能否说出专业术语，而在于它是否具备稳定的产品闭环。四季夏木的完成态标准包括：第一，用户上传图片后能够获得真实识别结果而非预设文本；第二，结果不是单一标签，而是带有风险等级、行动卡片、证据说明与图鉴入口的结构化返回；第三，知识页面能够通过全文检索、来源索引与治理模板支撑进一步查证；第四，灵化系统能够基于识别种类自动加载角色包并开启流式对话；第五，社区内容能够沉淀为知识资产；第六，系统具备任务队列、错误恢复、可访问性与行为分析等工程基础能力。
二、真实完成态全景与能力矩阵
当前版本的四季夏木已经从“页面原型 + 概念交互”升级为“前台可操作、后台可执行、AI 能力真实接入、结果可沉淀”的完整系统。项目的能力不再停留在单页演示，而是在多个模块之间形成统一的数据流与任务流：同一张识别图片可以进入识别任务、写入历史、触发行动卡片、关联图鉴检索、解锁灵化角色、带入对话上下文，甚至进入社区求助与后续知识沉淀。

（一）为什么说它已经不是“聊天页项目”
很多 AI 项目在展示时看起来内容丰富，实质却只有一个聊天框和若干静态页面。四季夏木的完成态刻意避免这种空心化：一方面，识别任务、图鉴检索、社区帖子、生图任务、对话记忆、行动卡片和用户统计都以独立数据对象存在，能够被记录、查询、更新和复用；另一方面，用户的每一个操作都不是孤立事件，而是能引发后续模块联动。正因为形成了这种任务级、对象级的联动，系统才具备了“产品”而不是“展示页”的属性。
（二）完成品的五条主线
第一条主线是诊断主线：从上传图片到识别结果、风险等级、行动建议与复查路径，构成实际问题解决闭环。
第二条主线是知识主线：从首页搜索到图鉴全文检索、详情页来源索引、治理模板与相关推荐，构成证据化学习闭环。
第三条主线是灵化主线：从识别种类到角色包选择、角色生成、流式对话与记忆延续，构成可持续互动闭环。
第四条主线是协作主线：从社区发帖到楼层讨论、带坐标标注的图片回复、已解决状态与知识沉淀，构成共建闭环。
第五条主线是成长主线：从识别、收藏、发帖、解答到勋章与行为分析，构成长期留存与激励闭环。
三、程序整体布局与跨端体验架构
在视觉与交互层面，四季夏木采用“移动优先、桌面增强、任务优先”的布局原则。也就是说，所有核心操作必须首先适配手机使用场景，因为校园中最常见的使用方式就是边走边拍、边看边处理；同时，桌面端通过固定信息栏与更大的内容容器，补足图鉴检索、社区阅读与数据管理时的效率需求。
未登录态的欢迎页、登录、注册与忘记密码页面全部采用全屏移动优先布局，视觉焦点集中在品牌图、校园背景和主操作按钮上，尽量减少干扰项，让用户在首次进入时能够迅速理解“这是一个帮助你解决植物问题的系统”。这部分页面的设计目标不是信息堆叠，而是让用户尽快完成进入与信任建立。
登录后，系统统一进入 PageHeader 顶部标题栏 + 内容区的主框架。桌面端采用左侧固定信息栏、右侧主内容区的结构：左栏承载项目卡片、主导航与今日提示，右侧承载具体页面功能；手机端则将导航收束为底部 5 Tab，即首页、图鉴、灵化、社区、我的，隐藏桌面侧边栏，以保证单手操作与首屏信息密度。
（一）全局组件与系统级体验
全局浮层：右上角常驻消息中心按钮，点击后以通知抽屉形式展示系统消息、任务完成提示、收藏变动与社区互动反馈。
Toast 消息提示：所有轻量反馈采用统一 Toast 组件，包括成功、失败、警告与处理中四类状态，避免页面跳转打断用户当前任务。
统一状态视图：Loading、Empty、Error 三类状态组件共享同一套视觉风格与交互策略。Loading 强调过程可感知，Empty 强调可恢复动作，Error 提供明确的重试或回退入口。
可访问性支持：系统提供“跳到主内容”链接、可见焦点样式、ARIA 标签、路由切换播报与键盘可操作路径，使页面不仅能“看起来完成”，也能在规范层面站得住。
（二）信息架构的组织逻辑
如果把四季夏木理解为一个面向校园的任务型应用，那么它的前台信息架构其实非常明确：首页负责聚合入口与风险提醒，AI 识别负责问题起点，图鉴负责体系化证据，灵化负责互动化理解，社区负责协作与求助，我的页面负责账户与成长，分析页负责演示和验证。每个模块各有重心，但都围绕“植物问题处理”这一核心目标展开，因此用户不会在系统中迷路。
（三）UI 设计的判断标准
设计上，项目刻意避免堆砌华丽视觉，而是强调可用性、一致性与场景适配。所有页面遵循统一的标题层级、间距体系、按钮语义、标签风格与风险色彩规范。首页强调引导，图鉴强调检索，灵化强调沉浸，社区强调阅读与回复效率；虽然模块气质不同，但它们共享同一套品牌语言。这种统一性能够显著提升答辩观感，因为评委会直接感受到它不是拼凑出来的多页面集合，而是一套完整设计系统。
四、页面系统与使用说明
本节按用户实际可见页面逐一说明系统功能，并将当前已经完成的真实后端能力接入进去进行统一表述。。
1. 欢迎页 /welcome
欢迎页承担的是品牌建立与场景引入的职责。页面以校园背景、程序 Logo 与品牌口号为视觉中心，主操作按钮为“进入探索”。在完成态中，这一页既是访客入口，也是登录状态判断点：若检测到用户已有有效会话，则可以直接跳转首页，减少重复登录摩擦。
欢迎页的意义不只是“好看”，它实际上承担了产品定位的第一句话：四季夏木关注的不是抽象 AI，而是校园环境中的植物健康问题。用户一进来就能知道自己来到的是一个有明确问题域、有清晰行动目标的系统。
2. 登录 /login
登录页采用顶部品牌区、校园图横幅、账号密码表单、忘记密码入口、注册入口与示例账号提示的组合布局。前端实时校验账号格式与密码长度，错误信息就地展示，不让用户在表单提交后才发现问题。
完成态下，登录页已经接入真实身份验证流程。用户输入凭证后，系统会完成账号校验、会话写入、用户资料拉取与导航恢复；若用户曾停留在某条图鉴或某次识别任务中，也可以在登录后回到上一次上下文。
3. 注册 /register
注册页保留四段式表单：昵称、账号、密码、确认密码。系统在前端执行必填、格式、强度与一致性校验，在后端执行账号唯一性与数据写入。注册成功后，页面自动返回登录页并预填账号，形成顺畅的新用户冷启动链路。
这一页对应的是平台长期运营的最小基础。它说明四季夏木不是临时演示，而是考虑到了用户账户体系、身份延续与长期使用场景。
4. 忘记密码 /forgot-password
忘记密码页保留账号、新密码、确认密码三段式表单，前端校验账号存在、新旧密码不同与强度合规，后端完成凭据更新。成功后回到登录页，形成闭环。
5. 首页 /home
首页是整个系统的任务中枢。页面顶部设置搜索条，可直接跳转图鉴检索；中部放置校园推荐 Banner 与四宫格快捷入口，分别指向 AI 识别、百科图鉴、社区与灵化；下部承载本月高警报与社区精选，用于展示校园中更值得注意的问题与高质量内容。
完成态下，首页并非静态菜单页，而是实时汇聚任务入口、热点内容与风险提示的综合看板。用户既可以从搜索条直接进入图鉴，也可以从四宫格进入识别或灵化；而本月高警报与精选内容则让首页具备“校园植物健康入口”的属性。
6. AI 识别 （智能鉴别）/identify
识别页沿用上传区、预览区、生成按钮、识别结果卡与最近识别列表的结构，但底层已经由 Mock 结果升级为真实图像识别流程。用户上传图片后，前端首先生成本地预览并创建识别任务，随后由后端写入任务队列，调用真实图像识别 API 进行检测。
识别完成后，系统返回的不再是单一名称，而是一组结构化结果：候选病虫害或异常类型、主判断结果、置信度、风险等级、识别标签、证据摘要、行动卡片以及图鉴关联入口。最近识别列表也不再只是前端演示数据，而是由真实任务记录组成，能够被回看、对比与再次进入详情处理。
7. 图鉴库 /encyclopedia
图鉴库页面保留搜索栏、类型切换、专业筛选区、结果网格卡片、收藏夹、最近浏览与空状态恢复按钮的整体布局。在完成态中，搜索逻辑已由简单筛选升级为全文检索 + 结构化过滤的组合模式。用户输入关键词时，系统会在名称、别名、寄主、症状、危害、治理方式与来源标签等多个字段中做统一检索和归一化处理。
专业筛选区覆盖分类、风险、排序、联网检索与重置等功能；结果卡片继续显示季节、风险、寄主、分类、收藏与加入对比；收藏夹和最近浏览则由真实数据驱动。用户筛选后点击卡片，即可进入图鉴详情，并在详情页看到来源索引、治理模板和相关推荐。
8. 图鉴详情 /encyclopedia/:id
图鉴详情页延续左侧图文详情、右侧关键指标摘要卡的双区布局。左侧聚合形态特征、症状危害、防治建议、放置建议与参考来源；右侧摘要卡用于快速展示风险等级、主要寄主、发生季节、处理优先级和是否建议隔离等关键指标。
完成态中，这一页新增了三类高价值内容。第一类是来源索引，明确展示条目背后的资料出处、来源类型与检索可信度；第二类是治理模板，将分散知识整理为可执行的处置模板；第三类是相关推荐，包括相似病害、相似虫害、同寄主问题和同季节高发项，使图鉴详情真正成为证据中枢，而不是一页静态百科。
9. 灵化互动 /spirit
灵化页是系统最具辨识度的模块，也是本次更新变化最大的一页：用户拍照或上传图片后，系统先进行真实智能鉴别；当识别出昆虫或植物病害后，页面解锁“生成灵化角色”；用户点击后，系统直接进入角色生成与上下分层灵化页，。
上下分层灵化页的上半部为幻化卡片，支持“灵化立绘 / 昆虫实拍图”切换；中间区域浮现专业标签，用于把识别结果、危害等级、寄主信息与关键症状可视化；下半部是视觉小说式对话框，底部提供 3 个叶片按钮——防治建议、生活习性、外貌特征——以及自定义输入框。用户既可以点击快捷问法，也可以直接开展连续对话。
10. 社区列表 /community
社区列表页保留搜索栏、状态筛选、结果卡片与发帖入口。完成态下，搜索不仅支持标题、正文与作者，还支持对回复内容的检索，因此用户能够从更细粒度的讨论里找到相似问题。卡片同时展示楼层数与最新楼层类型，让用户快速判断帖子活跃度与解决进度。
这一页的价值在于把零散个体问题聚合为共建知识场。对校园用户来说，它不仅是求助区，也是观察别人如何处理类似问题的学习场。
11. 社区发帖 /community/new
发帖页延续标题、详细描述、富文本预览、@/# 标签提取、图片上传与手机拍照上传的结构。用户输入内容后，系统提供 Markdown 语义展示与结构化预览，确保文字表达清晰可读；图片上传前可做压缩预处理，以兼顾体验与传输成本。
完成态中，发帖内容与标签会同步进入可搜索索引。也就是说，用户今天发的一次求助，不会只停留在当下讨论，而是有机会在未来成为别人的搜索入口。
12. 社区详情 /community/:id
社区详情页延续 1 楼主贴、楼层讨论区与发楼层面板的贴吧式结构。完成态支持楼主追问、普通回复、回复某层、上传图片以及病斑/虫体标注坐标。标注坐标使讨论从“说不清楚在哪里”变成“准确指出问题区域”，显著提高沟通效率。
当帖子被标记为已解决或高质量时，系统可以把其中的优质答案抽取为知识候选，进入后续图鉴补充或治理模板优化流程，从而形成社区到知识库的反向流动。
13. 我的成就 /me
“我的”页面包括个人卡、账号动作、资料编辑、四项统计与勋章墙。用户可以在此修改昵称与头像，查看发布、解答、收藏、识别等核心数据，并通过勋章墙看到个人成长路径。
这一页的设计重点不在于炫耀数据，而在于让用户知道：自己的使用行为会被系统看见，会形成进度，会产生价值。对校园项目来说，这一点有助于促进社团成员持续参与。
14. 行为分析 /analytics
分析页展示事件总览、事件明细与清空按钮，用于演示前端埋点结果和用户路径。当前版本延续本地存储展示方式，能够清楚呈现欢迎页进入、注册、登录、首页搜索、AI 识别、图鉴浏览、社区发帖、灵化生成与对话等关键动作。
五、关键业务闭环：从识别到行动、从学习到共建
真正打动评委的，往往不是单个功能点有多酷，而是系统能否把用户从起点顺利带到结果。四季夏木的设计重点就在于闭环：每一条用户路径都不是孤立页面跳转，而是围绕某个真实目标被组织起来。
（一）新用户冷启动路径
用户从欢迎页进入系统，完成注册、登录与基础资料建立。
进入首页后，通过搜索条或四宫格理解系统主能力。
第一次检索图鉴或第一次发布社区求助，完成从浏览者到使用者的身份转换。
系统通过消息提示、收藏夹、最近浏览和勋章进度，让用户形成再次返回的理由。
（二）智能识别处理路径
这是系统最核心的一条路径，也是评委最容易直接感知其价值的一条路径。用户在首页进入 AI 识别页，上传植物或虫体图片后，系统会立即创建识别任务，写入队列并返回处理中状态；任务完成后，用户看到的不只是“这是什么”，而是一套可以继续操作的结构化结果。

第四步，行动卡片引擎生成“立即处理 / 观察复查 / 隔离建议 / 图鉴查证 / 社区求助”等下一步动作。
第五步，识别记录写入最近历史，并自动关联图鉴详情与相关推荐。
（三）图鉴学习与查证路径
当用户对识别结果不够确定，或者希望做进一步学习时，会进入图鉴路径。全文检索与结构化筛选使用户可以按关键词、类型、风险、寄主和季节进行搜索；进入详情页后，通过来源索引、治理模板与相关推荐完成二次确认。相比“问一问就结束”的系统，四季夏木鼓励用户把 AI 输出与知识证据放在一起理解，从而建立更加可信的判断。
（四）灵化互动学习路径
灵化路径不是为了娱乐而独立存在，而是为了降低知识理解门槛。用户上传图片并完成智能鉴别后，系统根据识别结果解锁角色生成；角色生成完成后，用户在视觉小说式场景中通过快捷问题或自由输入进行连续追问。角色化表达将专业知识转译为更适合青少年接受的叙事语言，从而提升记忆度和参与感。
（五）社区协作与知识回流路径
当系统识别结果置信度不足、情境复杂或用户仍不放心时，社区就是最后一层协作网络。用户可以把识别图、处理过程和具体疑问发到社区，由其他用户分楼层回复；高质量回答不仅解决当前问题，也可以在后续被审核并沉淀为图鉴补充、治理模板或 FAQ。也就是说，社区不是知识的终点，而是知识增长的入口。
六、AI 识别中枢：真实图像识别 API、风险等级与行动卡片
四季夏木之所以具备“实战级”气质，很大程度上来自 AI 识别中枢的完成。这里的关键不是单纯把一个识别接口接上去，而是把识别结果组织成可调度、可审计、可落库、可继续使用的业务对象。完成态下，AI 识别页背后运行的是一套包含图片接收、任务创建、推理调用、结果标准化、风险计算、行动生成、图鉴关联与历史写回的全链路流程。
（一）识别任务对象化：从上传动作到业务对象
用户点击上传后，前端首先会完成本地预览、基础校验与媒体信息采集，然后调用后端接口创建识别任务。此时系统不会把“识别”视为一个瞬时行为，而是创建一条可追踪的 diagnosis task。该任务具有明确的状态流转：pending、queued、running、succeeded、failed。这样做的好处是，识别 API 即便存在延迟、限流或重试，也不会破坏前端体验；用户可以在页面上感知任务进行中，结果完成后也能够被完整追溯。

（二）真实图像识别 API 的接入策略
完成态下，后端已经接入真实图像识别 API。后端工作线程从队列中取出待处理任务，读取上传图像，调用识别服务，并将服务返回的原始结果归一化为系统内部统一结构。归一化的必要性很强，因为外部识别服务往往返回的是模型原始字段，而产品真正需要的是可以直接驱动界面的业务字段。
原始候选结果会被标准化为名称、类别、置信度、别名、症状标签与主要证据。
当识别结果包含多个候选时，系统会保留 Top-N 候选并生成主判断与备选判断。
当置信度不足时，系统不会强行装作确定，而是显式提示“低置信度 / 建议查证 / 建议社区求助”。
识别结果与图鉴条目建立映射关系，便于用户一键跳转到更细致的说明页。
（四）行动卡片：把 AI 输出翻译成可执行步骤
识别结果之所以容易“看完就忘”，是因为普通系统只告诉用户结论，不告诉用户怎么动手。四季夏木将风险结果进一步转化为行动卡片，每张卡片都是一个明确、可执行、可跳转的下一步。行动卡片不是附属文案，而是诊断页的核心交互对象。
立即处理卡：用于提示需要优先采取的操作，如移除病叶、隔离植株、减少浇水或避免触碰扩散源。
观察复查卡：给出建议的复查时间窗口与观察重点，例如 24 小时、48 小时或 3 天后重新检查叶片背面。
图鉴查证卡：一键跳转到匹配条目，帮助用户确认形态特征、传播方式和治理依据。
社区求助卡：当结果不确定或环境复杂时，鼓励用户带图发帖，提高问题解决成功率。
持续记录卡：将该植物加入关注对象，为后续对话和记忆提供稳定上下文。

（五）最近识别历史为什么重要
最近识别列表并不是一个附带功能，它承担了“结果可追溯”的责任。用户今天识别过的对象，明天需要复查时不能从零开始；系统必须能够记住它曾经是什么、当时风险如何、用户采取了哪些行动。真实完成态下，最近识别历史因此具有很强的业务价值：它是连续诊断、长期记忆和行为分析的入口。
七、图鉴知识引擎：全文检索、来源索引、治理模板与相关推荐
图鉴模块的定位并不是“把百科搬进来”，而是把识别后的不确定感转化为有证据支撑的理解过程。完成态下，图鉴已经从静态条目页升级为知识引擎：它不仅能被搜到，还能被筛选、被对比、被收藏、被引用，并能反向驱动行动卡片和灵化对话。
（一）全文检索：从关键词匹配到问题表达理解
过去图鉴页更多承担展示角色，而当前版本已经接入全文检索，支持对名称、别名、寄主、症状描述、危害表现、治理方式、来源标签和社区沉淀关键词进行统一索引。用户输入“叶片发黄”“白色小虫”“卷叶”“霉层”等生活化描述时，系统并不会要求用户先知道专业术语，而是通过归一化检索把这些表达映射到更接近的专业条目。
名称字段支持学名、常见名与别名联动检索。
症状字段支持生活化语言与专业术语混合命中。
风险与季节字段参与排序，帮助用户优先看到更相关的高风险结果。
搜索结果可与类型切换、分类筛选、风险筛选和排序规则同时使用。
（二）来源索引：让知识页面从“像真的”变成“有根据的”
AI 项目经常会被质疑“回答像模像样，但依据在哪里”。四季夏木在图鉴详情中增加来源索引，就是为了解决这一点。来源索引不只是列一串链接或书名，而是把条目所依据的资料类别、来源名称、出处片段、引用目的和可信度标签结构化展示。
来源索引支持按来源类型区分，如专业资料、公开科普资料、整理型知识条目等。
每条图鉴可以绑定多个来源项，用于分别支持形态特征、症状危害与防治建议。
来源索引既服务用户查证，也服务系统内部审核与后续更新。
（三）治理模板：把知识条目转成动作模板
治理模板是图鉴模块的重要升级。很多知识条目写得很完整，但用户仍旧不知道如何下手；治理模板则将分散的知识段落压缩成面向执行的结构。模板通常由观察项、立即处理项、环境调整项、复查节奏项与注意事项构成，可以直接驱动诊断页中的行动卡片，也能作为灵化对话时的知识骨架。

（四）相关推荐：让知识不是单点，而是网络
图鉴详情页新增相关推荐逻辑，用于在用户查看某一条目时同步展示相似病害、同寄主高发问题、同季节高频问题及治理方式相近的问题。这个设计非常重要，因为真实场景中用户往往不是一次就看准，系统必须帮助他建立“相关问题网络”。相关推荐不仅提升图鉴页停留价值，也提高了用户进一步学习和比对的效率。
（五）收藏、最近浏览与对比为什么有价值
收藏与最近浏览让图鉴从一次性阅读变成可持续参考；而加入对比功能，则能帮助用户在多个相似条目之间快速切换关键差异，例如症状位置、危害速度、寄主范围和处理重点。对于初学者来说，对比功能往往比长篇文字更容易建立判断。这说明图鉴模块已经不再是“能看”的内容页，而是“能用”的知识工具。
八、夏木对话中枢：提示词编排器、流式对话、种类驱动角色包与长期记忆
灵化互动之所以成立，不是因为多了一张二次元角色图，而是因为背后有一套能够稳定组织语气、知识、上下文和记忆的对话中枢。当前版本中，这个中枢被统一称为“夏木提示词编排器”。它的任务不是单纯给模型下命令，而是把系统策略、识别结果、图鉴知识、角色特征、用户历史与当前问题整合为一份结构化上下文，使每一次回答既有人格温度，也有专业依据。
（一）夏木提示词编排器的层级结构
夏木提示词编排器采用分层注入方式组织上下文，而不是把所有信息堆成一段超长 Prompt。这样做有两个好处：一是输出更稳定，二是后续更容易维护和审计。
系统策略层：定义安全边界、校园场景适配、输出风格约束与禁止事项。
角色人格层：定义“夏木”的表达语气、陪伴感、科普方式与鼓励性语言节奏。
种类驱动角色包层：根据识别出的种类自动加载对应的角色设定、用词偏好、知识重点与视觉关键词。
诊断上下文层：注入本次识别得到的名称、类别、置信度、症状标签、风险等级与行动建议。
检索知识层：注入图鉴摘要、治理模板、来源索引摘要与相关推荐。
记忆层：注入用户偏好、最近关注植物、未解决问题与历史对话摘要。
当前任务层：根据用户点击的快捷按钮或自由输入，决定本轮回答的直接目标。

（二）流式对话：让系统像在“实时陪你看问题”
当前版本已经接入流式对话，不再等到模型完整生成后一次性回填页面。用户在灵化页输入问题或点击叶片按钮后，系统会发起流式响应，逐段输出文本，前端同步显示加载态、首字到达、持续生成和结束状态。
流式输出降低等待焦虑，让用户在较长回答场景中保持参与感。
首字延迟、停止生成、重试生成等细节更接近真实产品体验。
流式内容在结束后会被结构化写入会话记录，便于后续记忆压缩与检索。
（三）种类驱动角色包：灵化不是随机扮演，而是由识别结果驱动
四季夏木中的角色并非“随便生成一个形象再聊天”，而是由识别结果直接驱动角色包选择。比如，识别出蚜虫、螨类、蛾类、叶斑病、霉层类问题时，系统会进入不同的角色包分支。每个角色包包含的不只是形象关键词，还包括话语节奏、比喻词汇、知识优先级、禁止误导列表、默认快捷问法和生图提示词参数。
虫害类角色包更强调繁殖速度、寄主关系、扩散路径与观察部位。
病害类角色包更强调症状演变、环境条件、湿度通风与治理节奏。
角色包会避免过度戏谑，保证既有亲和力，也不失专业感。
（四）长期记忆：把“这次提问”连接成“持续陪伴”
长期记忆是当前版本完成度很高的一项能力。没有记忆的系统，每一轮都像第一次见面；有了记忆，系统才能真正持续跟踪同一盆植物或同一个用户的学习轨迹。四季夏木的长期记忆并不是把所有原始消息机械堆积，而是按层级进行管理。
短期会话记忆：保留当前会话最近若干轮消息，用于维持即时连贯性。
问题案例记忆：记录用户最近关注过的植物、识别过的问题、是否已处理、是否待复查。
偏好记忆：记录用户更偏好的解释方式、是否喜欢简明回答、是否更关注防治或科普。
摘要记忆：将长会话压缩为结构化摘要，降低成本，同时保留关键事实。
（五）为什么这套对话中枢适合比赛答辩
评委通常会非常警惕“角色化是否牺牲专业性”。而夏木提示词编排器恰恰能回应这个问题：角色化不是为了掩盖知识，而是为了更好地组织知识。系统真正做的是把专业知识包裹在合适的表达方式里，把难懂内容转成可记忆内容，同时保持风险等级、治理模板与来源索引这些硬核骨架不丢失。这正是四季夏木与一般“二次元聊天壳”最大的区别。
九、多模型路由：云端 DeepSeek、本地 Ollama 与任务分流
四季夏木采用多模型路由而不是单模型硬顶所有任务。在线主力对话由硅基流动提供的 DeepSeek API 承担，原因在于它具备较好的综合理解与生成能力，适合承担一般图鉴解释、结构化问答、社区总结与人格化回复。对于本地敏感、弱网或离线兜底场景，系统预留 Ollama 路由，用于执行轻量问答、本地摘要与限定知识域下的对话任务。图像生成则单独走 ComfyUI 工作流，不与文本模型混用。
路由器的判断维度包括任务类型、延迟容忍度、内容敏感性、是否需要本地执行、是否需要图像输出、是否有高上下文历史依赖等。例如，用户点击“防治建议”时，系统优先走 DeepSeek 并附带诊断上下文与治理模板；若设备处于离线或本地优先模式，则改由 Ollama 使用压缩版提示词与本地缓存知识作答；若用户点击“生成示意图”，则直接创建 generation_job 并把文本提示、角色包与参考条件提交到 ComfyUI。
这种混合路由有三重意义：第一，性能与成本更平衡；第二，敏感与离线场景更可控；第三，未来扩容空间更大。对竞赛答辩而言，这说明项目对“AI 能力接入”不是一次性接 API，而是已经开始向真正的智能中台形态靠拢。
十、ComfyUI 工作流：角色立绘、病害示意图与科普视觉素材生成
ComfyUI 在四季夏木中承担的不是一个附属创意工具，而是一条被业务系统正式调用的生图链路。前端点击“生成灵化角色”“生成病害示意图”或“生成科普素材”时，并不是直接把一句自然语言发送给图像模型，而是创建一条 generation_job。后端根据任务类型、识别结果、角色包与输出用途映射出具体工作流 JSON，再向 ComfyUI 提交执行。任务结果通过轮询或回调方式回写至 generation_jobs 表，并同步关联到 media_assets 与 conversation_sessions。
典型的角色立绘工作流大致包含以下节点：CheckpointLoaderSimple 用于加载基础模型；CLIPTextEncode(positive) 编码正向提示词，CLIPTextEncode(negative) 编码负向约束；若需要参考用户上传图像，则通过 LoadImage + VAEEncode 或 IPAdapter / ControlNet 相关节点提取条件；随后由 EmptyLatentImage 或参考潜空间进入 KSampler 进行采样；最后经由 VAEDecode 解码并由 SaveImage 输出结果。若任务类型为病害步骤图或科普海报，还会加入文本版式占位、局部重绘、背景净化或 ControlNet 结构控制节点，以保证教学图像清晰可读。
在业务映射上，系统至少维护三类工作流模板。第一类是 persona_portrait_workflow，偏重角色立绘与视觉一致性；第二类是 pathology_explainer_workflow，偏重病斑位置、症状阶段与步骤图表达；第三类是 outreach_poster_workflow，偏重宣传海报与校园科普物料。角色包会向这些工作流注入 visual_keywords、mood_keywords、forbidden_elements 与 output_size 等参数，而识别结果会注入 pest_name、symptom_tags、risk_level、host_plant 与 reference_image_path 等条件。这样，生图不是“随机生成”，而是一次带有业务语义的任务执行。
十一、ComfyUI 接入细节与任务协议设计
为了让 ComfyUI 稳定接入产品系统，四季夏木在后端设计了统一的任务协议。generation_job 至少包含字段：job_id、job_type、workflow_name、prompt_payload、reference_images、status、queue_position、result_urls、error_message、created_by、linked_diagnosis_id 与 linked_conversation_id。前端只关心 job_id 与状态轮询，具体工作流细节由后端维护，避免前端直接耦合 ComfyUI 的节点结构。
一个典型的提交载荷可能包括 visual_profile（角色包视觉设定）、domain_facts（识别出的病虫害与寄主信息）、style_constraints（校园科普或二次元风格强度）、safety_rules（不允许的元素与输出限制）以及 output_spec（尺寸、比例、格式、数量）。后端会把这些业务字段编译为工作流所需的节点输入：正负提示词、参考图地址、种子、采样器、步数、CFG、分辨率与后处理节点参数。
当任务完成后，系统并不会直接把所有生图结果都发给用户，而是可以设置质量门控，例如去除明显失真、内容偏离或文本不可读的输出，再保留最佳候选写回媒体资产。这样既提高成品质量，也使答辩展示更加稳定。未来如需扩展到批量海报生成、社团宣传图生成或图鉴插图补全，只需要增加新的 workflow_name 与编译映射规则。

（二）角色包如何参与生图
种类驱动角色包在生图阶段主要承担三个作用。第一，提供形象生成所需的视觉关键词，例如颜色、质感、翅形、触角、病斑意象和植物寄主元素；第二，提供表达气质，例如温和陪伴型、知识讲解型、警示引导型；第三，提供限制项，避免生成与当前问题无关或误导性的视觉元素。这样，角色图并不是“漂亮就行”，而是需要在审美、辨识度与知识相关性之间取得平衡。
（三）生图完成后的页面组织
角色图生成完成后，用户看到的不是一张孤立图片，而是完整的灵化页：上半部分的幻化卡片用于切换角色图与真实图像，中间的专业标签用于强化知识锚点，下半部分的视觉小说对话框承载流式互动，底部叶片按钮提供低门槛提问入口。这个页面的设计逻辑非常清晰：先用视觉抓住注意力，再用标签建立专业定位，最后通过对话把用户带入深度理解。
十二、社区协作、个人成长与行为分析：让系统真正“活起来”
如果说识别与图鉴解决的是“系统自己能做什么”，那么社区、成就与分析解决的就是“用户参与后会发生什么”。这部分能力往往最能说明一个项目是否考虑过真实运营。四季夏木并没有把社区做成单纯留言板，而是把它设计为围绕植物问题展开的结构化讨论区；同时，又用成长反馈与行为分析把用户使用轨迹可视化，构成长期循环。
（一）社区不是附属功能，而是复杂问题的协作层
真实场景中，很多植物问题并不能被一次识别完全解决。图片拍摄角度有限、病症处于早期、环境因素复杂、多个问题叠加，都可能导致识别结果需要人工补充判断。社区的价值就在于，它为系统提供了一层“群体经验”的补足。用户在社区中发帖时，可以上传图片、添加描述、提取 @ 与 # 标签，并通过富文本预览确保表达清晰。其他用户则通过楼层回复、回复指定楼层与图片标注坐标来协助判断。
楼层式结构使讨论有上下文，楼主追问也更自然。
回复图片支持病斑 / 虫体标注坐标，让“这里有问题”变成精确指认。
帖子支持待解决 / 已解决状态筛选，便于用户优先查看更有帮助的内容。
帖子、回复与标签同时参与检索，使社区内容具备二次发现价值。
（二）社区到图鉴的知识回流
四季夏木的一个重要设计点，是让社区优质内容有机会回流知识库。当某个帖子出现高质量回答、清晰图示与稳定结论时，系统可以将其标记为知识候选，由管理员或维护者进一步整理进图鉴条目、治理模板或 FAQ。这种回流机制意味着，用户贡献不是消耗品，而是系统成长的素材。对于评委来说，这说明项目具备“自我增厚”的潜力。
（三）个人成就不是游戏壳，而是长期留存机制
“我的成就”页面中的个人卡、统计与勋章墙，不是为了把系统做得花哨，而是为了建立用户长期参与的内在动力。用户在发帖、解答、收藏与识别这些行为上的积累，都会反馈到统计数据和勋章进度上。这样一来，系统就不只是“解决问题”，还会让用户看到自己在共同维护校园植物健康。
发布统计强调用户的求助与分享行为。
解答统计强调共建价值，让回答行为被看见。
收藏统计强调知识积累过程。
识别统计强调与现实植物互动的频率。
（四）行为分析页为什么值得保留
很多比赛项目只在答辩时讲“我们很重视用户体验”，但并没有真正记录用户体验。四季夏木保留行为分析页，就是为了让这件事具象化。当前版本中，分析页主要展示前端本地存储的事件计数与时间明细，能够清晰还原新用户路径、智能识别路径和灵化演示路径。同时，在系统底层，服务端也可保留任务日志、接口耗时和错误记录，用于工程监控与迭代优化。
十一、技术架构与数据底座：前台体验、后台编排与任务化服务
四季夏木之所以能够把识别、图鉴、灵化、社区与成长串成一体，靠的不是前端页面堆叠，而是明确的分层架构。完成态下，系统已经形成“体验层—业务接口层—任务与编排层—数据层—AI 能力层—运维安全层”的完整结构。每一层都有自己的职责，不相互混用，也不会把复杂逻辑硬塞进前端。

体验层负责所有用户可见能力，包括未登录态页面、登录后主框架、识别页、图鉴页、灵化页、社区页、个人页与分析页。前端以 React + Vite 构建，强调页面切换速度、组件复用、状态可控与移动端优先适配。PageHeader、Toast、通知抽屉、统一状态视图与底部 Tab 等都属于该层的通用 UI 基建。

（三）任务与队列层：让耗时操作可控
识别、生图和部分知识处理任务都不是适合在请求线程里硬等完成的动作，因此系统引入任务队列。任务队列的价值在于解耦页面等待与后端执行：前端只需要拿到任务 id 和当前状态，就可以通过轮询、推送或重进页面继续跟踪结果；后端则可以在更稳定的节奏下执行外部 API 调用、重试失败任务并记录全过程。
识别任务队列用于处理图像识别 API 调用、归一化与风险计算。
生图任务队列用于处理 ComfyUI 工作流提交、结果轮询、质量筛选与落库。
知识补全任务可用于处理来源索引整理、相关推荐计算或社区内容沉淀。
队列层让系统可以面对延迟、限流和偶发失败，而不会让页面直接崩掉。

在数据底座方面，四季夏木为 Supabase + Postgres + Storage 组合思路。它的优势在于既能承载账号体系、结构化数据与媒体存储，又方便做权限策略与后续扩展。完成态下，系统里的关键对象几乎都能在数据层找到对应实体，而不是散落在前端临时状态中。

AI 能力层：识别、对话与生图三条服务链
AI 能力层可以被拆成三条链路。第一条是图像识别链，负责看懂图片并给出结构化判断；第二条是对话链，负责在夏木编排器约束下生成流式回复；第三条是生图链，负责依据角色包与识别上下文生成灵化图像。这三条链虽然能力不同，但在系统中都被任务化、结构化地封装，因此它们可以在一套统一的产品逻辑里协同工作。
（六）统一 API/BFF 与任务队列设计
前端并不直接面对多个异构 AI 服务，而是通过统一 API/BFF 层访问系统能力。这一层负责路由、鉴权、参数标准化、结果归一化与错误恢复。它屏蔽了图像识别 API、DeepSeek API、Ollama、本地 ComfyUI 与 Supabase 的差异，使前端始终面对稳定的业务协议，例如 /api/identify/tasks、/api/chat/stream、/api/spirit/generate、/api/encyclopedia/search、/api/community/posts 等。
耗时操作全部任务化，是系统稳定性的关键。识别、生图、知识索引构建甚至部分摘要任务，都通过队列执行。前端拿到任务 ID 后，可以选择轮询、SSE 或 WebSocket 获取状态；后端 Worker 则可以进行重试、限流、失败回滚与耗时统计。状态流转通常包括 pending、queued、running、succeeded、failed 五段，必要时还可加入 canceled。
这种设计能很好地解释“为什么页面不会因为一个慢任务直接卡死”“为什么失败后可以重试”“为什么历史记录能被再次打开”。对评委而言，这些问题往往决定项目是否只是一个漂亮的前端壳。四季夏木之所以更接近真实产品，恰恰因为这些耗时能力被做成了工程化对象。
十二、Vibe Coding 研发模式与 Superpowers 工程化流程
本项目采用 Vibe Coding 研发模式，但并不是无约束地“让 AI 帮我写完一切”。在四季夏木中，Vibe Coding 的含义更接近“高速度生成 + 强约束收束”。Codex 负责快速生成页面骨架、接口定义、数据模型、测试脚本与样板逻辑；Stitch 负责提供原型语言、布局约束与设计 Tokens；Superpowers 则作为工程工作流，负责把这些生成能力串起来，防止系统因高速迭代而失控。
这里所说的 Superpowers，可以理解为一套面向 AI 研发协作的强化工作机制，包括：需求拆解、页面意图说明、接口契约先行、测试要点先列、生成后审查、预览部署、问题回流、样式归一与文案统一。它的作用不是替代开发，而是给 AI 生成与人工判断之间加上一层“质量闸门”。因此，项目在保持快速出稿能力的同时，仍能保证页面结构一致、数据口径一致、命名一致和交付口径一致。
在实际执行中，常见流程是：先由产品说明页面目标、状态和异常分支；再由 Codex 生成 React 页面、接口调用与类型定义；接着根据 Stitch 的 Tokens 统一视觉与组件；之后编写 API mock 与真实接入逻辑；再由测试脚本验证上传、识别、流式对话与发帖路径；最后通过 GitHub 提交、预览部署与回归检查完成交付。这种流程让“代码生成”从灵感行为变成了稳定产线。且项目代码已开源托管于 GitHub 平台，通过版本控制与持续集成机制支持团队协同开发与功能迭代，并与 Vercel 平台联动，后续会从阿里云，腾讯云购买域名部署。
十五、代码层面的 Superpowers 实践方式
在代码层面，Superpowers 体现为一套明确的约束模板。前端组件生成时，优先使用统一的 PageHeader、ActionCard、RiskBadge、AsyncPanel、EmptyState 等通用部件，避免一页一个写法；接口层采用 TypeScript 类型定义与 Zod 校验，保证前后端字段口径统一；状态管理上，将识别任务、会话流、社区表单与个人统计拆分为独立 store，降低跨页耦合。
在后端与脚本侧，Superpowers 体现为先写任务协议，再写实现。也就是说，在真正接 DeepSeek、Ollama 或 ComfyUI 之前，先确定 request/response schema、status enum、error code、retry policy 与 log payload。这样，即便后续替换底层供应商，前端与业务层几乎不需要大改。对于 AI 项目来说，这一点非常关键，因为模型与服务经常会变，而产品接口必须尽量稳定。
此外，测试脚本也被纳入生成链路。例如识别任务会有 smoke test 验证图片上传、状态流转与结果字段完整性；流式对话会验证首字延迟、结束标志与会话写入；社区发帖会验证标签抽取、图片上传与搜索索引命中。也就是说，Superpowers 不是某一段“炫技代码”，而是一种贯穿需求、实现、验证与部署的系统化能力。

十七、安全、可访问性、观测与运营可持续性
一个真正可用的校园 AI 系统，不能只讲模型能力，还必须对安全、可访问性、异常恢复和运营可持续性负责。四季夏木在这些方面的设计，恰恰构成了它区别于“演示型项目”的工程底色。
（一）安全边界：内容、权限与输出的三层控制
安全策略主要分布在三个层面。第一层是内容层，对社区发帖、回复、对话输出与生图结果进行基本合规约束，避免明显不适宜内容进入校园场景；第二层是权限层，通过账号状态、访问策略与数据权限避免用户越权访问他人资料和媒体；第三层是输出层，通过夏木编排器与结构化知识上下文减少模型随意发挥。
对话系统优先从图鉴知识、治理模板和来源摘要中组织回答，降低无依据扩写。
高风险问题会优先输出行动建议和查证建议，而不是模糊安慰式回答。
媒体访问可使用受控链接与分桶策略，避免资源随意暴露。
关键任务与错误会进入日志，方便回溯。
（二）可访问性：让系统对更多人真的可用
可访问性不是锦上添花，而是完成态系统应有的底线。四季夏木已经在全局层面加入“跳到主内容”、可见焦点、ARIA 提示与路由变更播报，同时在表单页、列表页和对话页保持较高的可点击面积、清晰的状态区分与一致的消息反馈。这些设计会直接提升比赛答辩时的专业感，因为它说明团队不仅关注视觉效果，也理解真实产品的规范。
（三）系统观测：知道哪里慢、哪里错、哪里被使用
完成态系统已经具备基础的可观测思路。前台通过分析页展示用户路径与操作事件，后台则可以记录识别任务耗时、生图任务耗时、对话首字延迟、接口错误率与任务失败原因。观测的价值不只在排错，更在于支撑迭代：例如首页哪个入口最常被点击，用户在图鉴页停留多久，灵化页最常点哪个叶片按钮，这些都能反过来指导产品调整。
十四、创新价值、答辩亮点与结论
（一）本项目最核心的创新，不是某一个功能，而是“结构性完成”
四季夏木真正有竞争力的地方，不在于单独拥有图像识别、全文检索、角色生成或社区中的某一个，而在于它把这些能力组织成了一套统一系统。识别结果可以进入风险等级和行动卡片；行动卡片可以跳到图鉴；图鉴可以用来源索引和治理模板支撑查证；识别种类可以驱动角色包、生图任务与流式对话；社区高质量内容又可以反哺知识库；个人统计与行为分析则为长期使用提供反馈。这种结构性完成，正是项目从“功能堆叠”走向“产品闭环”的关键。
（二）四季夏木有哪些最能打动人的点
它解决的是校园里真实存在、又容易被忽视的问题，题目本身有现实牵引力。
它不是停留在概念层，而是已经接入真实图像识别 API、真实对话 API 和完整生图服务。
它把识别结果转化为风险等级、行动卡片和治理模板，说明项目关注的是结果落地，而不只是模型炫技。
它把灵化角色建立在识别种类、角色包和长期记忆之上，说明角色化不是表面包装，而是产品策略的一部分。
它有社区、有知识沉淀、有行为分析、有可访问性与部署链路，说明工程意识完整。
（三）结语
从欢迎页到首页，从识别页到图鉴页，从灵化页到社区页，再到个人成长与行为分析，四季夏木已经形成了一套彼此联通的完成品结构。更重要的是，这套结构并不是靠概念描述硬撑起来的，而是由真实识别、真实对话、真实生图、全文检索、来源索引、治理模板、任务队列、角色包与长期记忆等能力共同支撑。
